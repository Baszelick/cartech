# CarTech AI Development Guide

> Этот документ обязателен для любого AI-агента, работающего над проектом CarTech.

---

# Project Overview

CarTech — Enterprise-система управления автомобильной площадкой дилерского центра.

Проект создается как максимально приближенный к коммерческой разработке.

Основная цель проекта:

- высокая читаемость кода;
- масштабируемость;
- единый стиль разработки;
- строгая архитектура;
- отсутствие "магии".

# General Principles

Всегда придерживайся следующих правил.

## SOLID

Следовать принципам SOLID.

---

## KISS

Не усложнять решение без необходимости.

---

## DRY

Не дублировать код.

Если логика повторяется три раза — подумать о выделении отдельного метода.

---

## Readability First

Читаемость кода важнее краткости.

Предпочтение отдается понятному коду.

---

# Architecture

Используется трехслойная архитектура.

```
Controller

↓

Service

↓

Prisma
```

Controller никогда не содержит бизнес-логики.

Controller отвечает только за:

- получение запроса;
- вызов сервиса;
- возврат ответа.

---

Service содержит всю бизнес-логику.

---

Prisma используется только внутри сервисов.

---

# Backend Rules

## DTO

Использовать DTO всегда.

Даже если объект состоит из одного поля.

---

## Response DTO

Никогда не возвращать Prisma Entity напрямую.

Использовать Response DTO.

---

## Validation

Использовать class-validator.

Не выполнять ручную проверку данных там, где это можно сделать через DTO.

---

## Transactions

Если операция изменяет несколько таблиц —

использовать

```
prisma.$transaction()
```

---

## Services

Метод сервиса должен выполнять одну бизнес-операцию.

Если метод становится длиннее 40–50 строк —

разбить его на private методы.

Пример.

```
create()

↓

validate()

↓

createArrival()

↓

createCars()

↓

getArrival()
```

---

## Private Methods

Предпочитать множество небольших private методов одному большому.

---

## Database

Использовать только Prisma.

SQL вручную не писать без необходимости.

---

## Migrations

Все изменения схемы проходят через миграции Prisma.

Не изменять структуру базы вручную.

---

# Frontend Rules

Использовать только Standalone Components.

NgModules запрещены.

---

Использовать Signals как основной механизм управления состоянием.

RxJS использовать только там, где он действительно необходим.

---

Не использовать any.

---

Использовать строгую типизацию.

---

# UI Library

Использовать только компоненты библиотеки.

Разрешено:

- ct-button
- ct-input
- ct-select
- ct-date-picker
- ct-icon
- ct-form-field

Не использовать обычные HTML элементы управления в бизнес-формах.

Запрещено:

```
<input>

<select>

<textarea>

<button>
```

если существует соответствующий компонент библиотеки.

---

# Styling

Использовать только SCSS.

Использовать дизайн-систему проекта.

Все размеры брать через CSS Variables.

Например

```
var(--space-16)

var(--radius-8)

var(--text-14)
```

Не использовать magic numbers.

---

# Naming

Использовать понятные имена.

Предпочитать

```
createArrival()

validateBrand()

nextBatteryCheck()
```

вместо

```
do()

run()

execute()

process()
```

---

# Code Style

Предпочитать ранний выход.

```
if (!user) {
    return;
}
```

вместо больших вложенных условий.

---

Не создавать глубокую вложенность.

Максимум 2–3 уровня.

---

# Business Logic

Вся бизнес-логика находится только в сервисах.

Никогда не помещать бизнес-логику:

- в Controller
- в Prisma
- в DTO

---

# Comments

Комментарии писать только если логика действительно сложная.

Не комментировать очевидный код.

Плохой пример

```ts
// Создаем пользователя
createUser();
```

Хороший пример

```ts
// Следующая проверка АКБ всегда назначается через 30 дней
```

---

# Performance

Предпочитать

```
createMany()
```

вместо множества отдельных create().

Использовать include повторно через константы.

Избегать N+1 запросов.

---

# Error Handling

Использовать стандартные исключения NestJS.

Например

- BadRequestException
- NotFoundException
- ConflictException
- ForbiddenException
