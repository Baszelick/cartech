# CarTech Business Rules

> Документ описывает основные бизнес-правила системы.
>
> Любое изменение логики приложения должно соответствовать данным правилам.

---

# Основная идея проекта

CarTech — система управления автомобилями дилерского центра.

Каждый автомобиль проходит полный жизненный цикл:

Приемка

↓

Хранение

↓

Предпродажная подготовка (ПСО)

↓

Контроль аккумулятора

↓

Выдача клиенту

---

# Жизненный цикл автомобиля

Автомобиль может существовать только после поступления автовоза.

Автомобиль не может существовать самостоятельно.

```
Arrival

↓

Car
```

---

# Приемка автомобилей

Приемка происходит автовозами.

Один автовоз может содержать любое количество автомобилей.

Автовоз без автомобилей создать невозможно.

Все автомобили одного автовоза должны сохраняться одной транзакцией.

Если хотя бы один автомобиль не удалось сохранить —

не сохраняется весь автовоз.

---

# VIN

VIN является уникальным идентификатором автомобиля.

В системе не может существовать два одинаковых VIN.

---

# Локации

Локация объединяет несколько площадок.

Пример

```
Х1

    └── Х1

В27

    ├── В27

    └── Поле
```

Количество локаций не ограничено.

---

# Площадки

Площадка принадлежит только одной локации.

На площадке разрешены только определенные бренды автомобилей.

Если бренд запрещен для площадки —

автомобиль создать нельзя.

---

# Бренды

Бренд хранится отдельно.

Бренд может иметь множество моделей.

Пример

```
Jetour

    X70

    X90

    Dashing
```

---

# Модели

Каждая модель принадлежит только одному бренду.

Автомобиль хранит ссылку только на модель.

Бренд определяется через модель.

Поэтому Brand никогда не хранится в таблице Car.

---

# Цвета

Цвет хранится отдельно.

Используется справочник цветов.

Свободный ввод цвета запрещен.

---

# Автомобиль

Автомобиль всегда принадлежит:

- одной площадке;
- одной модели;
- одному цвету;
- одному поступлению.

---

# Статусы автомобиля

Автомобиль может находиться только в одном статусе.

```
ARRIVED

↓

PSO

↓

READY

↓

ISSUED
```

Другие переходы запрещены.

Например

```
ARRIVED

↓

ISSUED
```

невозможен.

---

# ПСО

После завершения ПСО записывается дата выполнения.

Повторное прохождение ПСО не создается.

---

# Проверка аккумулятора

У автомобиля хранится история проверок аккумулятора.

Каждая проверка содержит:

- дату;
- напряжение;
- комментарий.

После каждой проверки автоматически рассчитывается следующая дата проверки.

По умолчанию следующая проверка назначается через 30 календарных дней.

---

# Выдача автомобиля

После выдачи автомобиля:

- записывается дата выдачи;
- статус становится ISSUED.

Выданный автомобиль не должен отображаться среди автомобилей в наличии.

---

# Удаление данных

Удаление автомобилей запрещено.

Удаление поступлений запрещено.

Используется только изменение статусов.

История должна сохраняться.

---

# Справочники

Все справочники являются динамическими.

Добавление новых:

- брендов;
- моделей;
- площадок;
- локаций;
- цветов

не должно требовать изменения кода приложения.

Все изменения выполняются через административный интерфейс.

---

# Архитектурные правила

Все бизнес-правила реализуются исключительно в сервисах Backend.

Контроллеры не содержат бизнес-логики.

DTO используются только для передачи данных.

Prisma используется только как слой доступа к данным.

---

# Основные принципы

При разработке всегда соблюдать следующие правила.

- История важнее удаления.
- Один автомобиль принадлежит только одному поступлению.
- Все сложные операции выполняются транзакциями.
- Бизнес-логика важнее структуры базы данных.
- API должно отражать бизнес-процесс, а не структуру таблиц.
- Любое изменение должно сохранять целостность данных.

---

# Главная цель

CarTech моделирует реальные процессы автомобильного дилерского центра.

Любая новая функциональность должна соответствовать существующим бизнес-процессам и не нарушать жизненный цикл автомобиля.
