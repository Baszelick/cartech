# ADR-001

## Использование собственной UI-библиотеки

### Статус

✅ Принято

### Проблема

Использование компонентов различных UI-библиотек приводит к разному поведению элементов, усложняет поддержку и затрудняет изменение дизайна.

### Решение

Создать собственную UI-библиотеку компонентов.

Все элементы интерфейса строятся только на собственных компонентах.

Например:

- ct-button
- ct-input
- ct-select
- ct-date-picker
- ct-form-field
- ct-icon

### Причина

- единый внешний вид;
- полный контроль поведения;
- отсутствие зависимости от сторонних UI Kit;
- возможность быстро менять дизайн всей системы.

### Последствия

Любая новая форма должна использовать только компоненты библиотеки.

Использование обычных HTML input, select и textarea запрещено.

# ADR-002

## Использование Angular Signals

### Статус

✅ Принято

### Проблема

Большая часть состояния приложения локальна и не требует полноценного глобального Store.

### Решение

Основным способом хранения состояния являются Angular Signals.

RxJS используется только там, где необходима работа с потоками данных.

### Причина

- меньше шаблонного кода;
- проще читать;
- выше производительность;
- современный подход Angular.

### Последствия

Signals используются по умолчанию.

NgRx подключается только при реальной необходимости.

# ADR-003

## Автомобиль хранит только Model

### Статус

✅ Принято

### Проблема

Бренд можно хранить двумя способами.

Вариант 1

```
Car

↓

brandId

↓

modelId
```

Вариант 2

```
Car

↓

modelId

↓

Brand определяется через Model
```

### Решение

Использовать второй вариант.

Car хранит только modelId.

Brand определяется через связь модели.

### Причина

Исключается дублирование данных.

Невозможно получить ситуацию:

```
Brand = Jetour

Model = BMW X5
```

### Последствия

Brand никогда не хранится в таблице Car.

# ADR-004

## Приемка автомобилей выполняется одной транзакцией

### Статус

✅ Принято

### Проблема

Во время приемки необходимо создать:

- Arrival;
- несколько Car.

При ошибке создания одного автомобиля данные могут оказаться неконсистентными.

### Решение

Использовать Prisma Transaction.

```
Arrival

↓

Cars
```

создаются внутри одной транзакции.

### Причина

Сохранение целостности данных.

### Последствия

При любой ошибке откатывается вся операция.

# ADR-005

## Все справочники являются динамическими

### Статус

✅ Принято

### Проблема

Количество брендов, моделей, площадок и цветов со временем изменяется.

Изменение этих данных не должно требовать изменения кода приложения.

### Решение

Использовать отдельные таблицы:

- Brand
- CarModel
- Color
- Location
- Site

### Причина

Расширяемость системы.

### Последствия

Добавление новых справочников выполняется через административный интерфейс.

# ADR-006

## Один сервис — одна бизнес-операция

### Статус

✅ Принято

### Решение

Сервисы описывают бизнес-процесс.

Пример

```
create()

↓

validate()

↓

createArrival()

↓

createCars()

↓

getArrival()
```

### Причина

- высокая читаемость;
- простота тестирования;
- простота сопровождения.

### Последствия

Методы длиннее 40–50 строк должны разбиваться на private методы.

# ADR-007

## Prisma используется только в сервисах

### Статус

✅ Принято

### Решение

Controller никогда не обращается к Prisma.

Архитектура:

```
Controller

↓

Service

↓

Prisma
```

### Причина

Разделение ответственности.

### Последствия

Вся бизнес-логика находится исключительно в сервисах.

---

# ADR-008

## DTO являются контрактом API

### Статус

✅ Принято

### Решение

DTO описывает внешний контракт API.

DTO не обязаны повторять структуру таблиц Prisma.

### Причина

Контракт API и структура хранения данных решают разные задачи.

### Последствия

Изменение базы данных не должно автоматически менять API.

# ADR-009

## История важнее удаления

### Статус

✅ Принято

### Решение

Основные сущности системы не удаляются.

Используется изменение статусов.

### Причина

Необходимо сохранять полную историю движения автомобиля.

### Последствия

Удаление Arrival и Car запрещено.
